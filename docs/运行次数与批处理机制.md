# ShellLink 运行次数与批处理机制

为了满足用户对同一个任务执行“多次抽卡（如连续跑 10 张图）”的需求，我们在工具栏引入了“循环运行次数”输入框。本文档解析该机制是如何从前端配置转化为工作流队列的。

## 一、 机制运作流程

整个批处理的编译和发送由 `action_run_executor.js` 负责，主要分为以下三步：

### 1. 基础任务编译器 (Task Compilation)

当用户点击“运行”或“运行全部”时，系统首先调用 `buildTasksQueue(isRunAll)`。
它会根据当前的 UI 状态（是否有选中的模块、选中的卡片），编译出一个**“基础队列 (Base Queue)”**。

- 基础队列的结构为一个数组，每个元素包含要执行的 `cardId` 和对应的 `previewAreaIds`。
- 例如，用户选中了卡片 A 和卡片 B，基础队列即为：`[ {Card A}, {Card B} ]`。

### 2. 队列倍增与引用的斩断 (Queue Multiplication)

系统会读取工具栏中 `#sl-run-batch-count` 输入框内的数字（如 N 次）。
接着，系统会执行一个 `for` 循环，将基础队列复制 N 遍拼接在一起：

```
const batchCount = getBatchCount();
let finalQueue = [];
for (let i = 0; i < batchCount; i++) {
    // 【核心细节】：使用展开运算符 {...task} 断开对象引用
    finalQueue = finalQueue.concat(baseQueue.map(task => ({ ...task })));
}
```

- **为什么要断开引用？**
如果直接 `push` 原对象，一旦后端在执行第一个任务时通过注入器修改了某些参数，后续的所有任务都会受到污染。使用深拷贝展开，保证了每个批次的任务都是独立的执行实例。
- 如果基础队列是 A 和 B，次数是 3，最终队列为：`[A, B, A, B, A, B]`。

### 3. 排队执行引擎 (`ShellLink.executeTasks`)

最终的 `finalQueue` 被移交给 `api_injector.js` 中的异步执行引擎。

1. **注入进度条**：引擎遍历整个队列，在 UI 上为所有涉及到的卡片显示 `5%` 的进度条预热。
2. **异步队列派发**：使用一个 `for` 循环，逐个向 ComfyUI 的原生 `app.queuePrompt(0, 1)` 发起请求。
3. **拦截与绑定**：每发送一个请求，原生的 `queuePrompt` 就会返回一个在服务器端生成的唯一 `prompt_id`。我们在 API 注入器中，将这个 `prompt_id` 与我们派发的任务信息死死绑定在 `window._slTaskMap` 中。

## 二、 报错阻断机制

在连续发送 N 个任务时，如果工作流的连线存在问题或者必填参数缺失，第一发请求就会被前端图谱校验器（Graph Validator）拦截报错。
此时，`executeTasks` 引擎的 `try...catch` 机制会立刻捕获异常：

- **终止发包**：直接 `break` 退出循环，不再发送后续的无用请求。
- **状态清退**：将当前失败卡片的进度条标记为红色错误（`error` 类），并将队列中其他还没来得及发的任务进度条瞬间归零清空，防止界面卡死。
