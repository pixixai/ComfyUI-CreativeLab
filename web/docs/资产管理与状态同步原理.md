# Creative Lab (CLab) 资产管理与状态同步原理解析

本文档详细阐述了 CLab 插件在图像/视频生成后的物理文件流转机制、工作流保存与读取原理，以及前端两项高级维护功能（清理失效记录、重新同步记录）的底层逻辑。

## 一、 运行后保存文件的机制 (Save Flow)

当用户点击“运行”并等待 ComfyUI 节点出图后，CLab 为了确保数据永久不丢失，在后台执行了一套极其严密的**“截胡与状态回写”**闭环流程：

1. **事件拦截 (前端)**：
通过 `api.addEventListener("executed")` 监听 ComfyUI 原生出图事件。节点生成完毕时，前端会捕获到刚生成的临时文件（存储在 `temp` 目录，随开即删）或常规输出文件。
2. **物理归档 (后端转移)**：
前端静默调用自定义的 Python 接口 `/clab/copy_temp_asset`。后端接到指令后，将 `temp` 或常规 `output` 目录下的媒体文件，**复制并重命名**（如 `pix_01.png`）到专属的永久归档目录 `output/CLab`（或您在设置中自定义的归档文件夹）中。
3. **内存回写 (状态闭环)**：
后端返回归档后的“永久物理路径”。前端接收到新路径后，主动去当前模块的 `area.history` 数组中查找并**替换掉**短暂的 `temp` 临时路径。
4. **硬编码落盘 (持久化)**：
路径更新后，立即调用 `StateManager.syncToNode(app.graph)`，将整个面板庞大的 `state`（包含了所有卡片的历史记录路径数组）序列化为 JSON 字符串，并**强制写入**到 ComfyUI 画布上的 `CLab_SystemConfig`（配置锚点）节点内。
5. **跟随工作流保存**：
当用户在 ComfyUI 中点击原生的“保存 (Save)”时，由于 `CLab_SystemConfig` 节点内包含了这段 JSON 文本，所有的资产索引路径便作为节点参数，被**永久保存**在了 `.json` 工作流文件中。

## 二、 读取文件的机制 (Load Flow)

当用户重新打开浏览器、重启 ComfyUI，或拖入一个已保存的工作流文件时，面板数据的复原流程如下：

1. **节点唤醒 (反序列化)**：
当 ComfyUI 原生系统把 `CLab_SystemConfig` 节点重新渲染在画布上时，插件的生命周期钩子会调用 `StateManager.loadFromNode(graph)`。
2. **内存重载**：
插件从该节点的隐藏输入框中提取出之前保存的巨大 JSON 字符串，解析成 JavaScript 对象，并**完整覆盖**当前的 `state.cards` 前端内存（几百张历史记录的软链接瞬间回归）。
3. **视图重绘 (Render)**：
触发全局自定义事件 `clab_render_ui`，面板根据载入的内存数据，重新生成所有的任务卡片、网格视图以及媒体标签（`<img>` / `<video>`）。
4. **物理拉取请求**：
浏览器根据重绘出的媒体标签（如 `<img src="/view?filename=pix_01.png&subfolder=CLab">`），向 ComfyUI 服务器发起普通的 HTTP 请求。服务器直接从本地硬盘的对应归档文件夹中读取物理文件并展示。

*(注：如果此时物理文件已在本地被删，服务器会返回 `404` 错误，媒体便会触发失效兜底逻辑显示为占位符。)*

## 三、“清理失效记录”的运行原理

在早期的设计中，检查文件是否存在需要向 Python 后端发送数据让后端校验。重构后的版本采用了**纯前端高并发试探**的架构，速度更快且无需注册新的后端接口。

1. **精准扫描目标**：
当在某个输出模块上点击“清理失效记录”时，系统仅收集该模块内部 `area.history` 数组中的所有媒体链接。
2. **轻量级探针 (HEAD Request)**：
遍历收集到的 URL，使用 JavaScript 的 `fetch(url, { method: 'HEAD', cache: 'no-store' })` 发起极其轻量的 HTTP 请求。
    - `HEAD` 方法：告诉服务器“我只要状态码，不要下载真实的图片数据”，极大地节省了网络带宽。
    - `cache: 'no-store'`：强制浏览器**绕过本地残影缓存**，必须去问服务器这个文件还在不在。
3. **状态裁决**：
如果服务器（硬盘）上文件已被删除，会明确返回 `404 Not Found`。探针捕获到 404 后，将该 URL 标记为“死链”。
4. **内存剔除**：
探针检查完毕后，使用 `splice` 将所有被标记为死链的 URL 从该模块的 `history` 数组中永久切除，并触发局部重绘同步保存最新状态，彻底清扫冗余数据。

## 四、“重新同步记录”的运行原理

这个功能是为了应对用户通过外部工具（如 Photoshop 覆写了文件）或资源管理器直接修改了物理文件，但浏览器界面死活不更新的痛点。它利用了经典的**“时间戳破除缓存 (Cache-Busting)”**技术。

1. **缓存陷阱**：
浏览器非常“懒惰”。如果 `<img src="pix_01.png">` 的 URL 字符串没变，哪怕硬盘里的文件已经被 PS 改得面目全非，浏览器依然会从内存里把旧的画面掏出来展示（俗称残影）。
2. **生成时间戳**：
当点击“重新同步记录”时，系统获取当前极其精确的毫秒级时间戳 `const now = Date.now()`。
3. **强制 URL 突变**：
遍历当前选中模块的所有历史 URL 和封面 URL，利用原生的 `URLSearchParams`，在链接末尾强行追加或替换一个无意义的查询参数，例如从：
`/view?filename=pix_01.png`
变成了：
`/view?filename=pix_01.png&t=1700000000000`
4. **强刷物理拉取**：
因为 URL 字符串发生了物理层面的改变，浏览器认为这是一个“全新的网络资源”，从而**彻底放弃本地残影缓存，强制向 ComfyUI 服务器发起全新的拉取请求**。
    - 如果文件被 PS 修改了，最新的画面就会瞬间刷新到面板上。
    - 如果文件被删除了，浏览器会立刻收到 404，从而真实地反映出图片丢失的裂开状态（随后便可接力使用“清理失效记录”一键扫除）。
